Designing a blog post page that still feels like part of the site

This template is built to keep the high-contrast, motion-led identity of the site while shifting the focus to readability. The point is not to make the article plain. It is to make the reading experience strong enough that the styling supports the words instead of competing with them.

Why this template exists

The list page is good at discovery, but it is not designed for sustained reading. An individual post page needs a different job. It has to keep navigation obvious, maintain the visual language of the rest of the portfolio, and still make long paragraphs easy to read without fatigue.

That changes the priorities. The title can still be bold and the hero image can still feel cinematic, but the content column has to calm down. The line length is narrower, the spacing is more generous, and the motion is saved for moments where it adds orientation rather than noise.

When you duplicate this page for a real post, the main thing to preserve is the structure. Change the copy, metadata, images, and related links in HTML, but leave the class names and shared asset includes alone so the styling and interactions keep working.

Designing for reading rather than browsing

The core article column uses Inter rather than the display font because extended reading needs rhythm more than attitude. Headings still use the site display type to keep the page recognisably yours, but the body copy steps back. That one change usually does more for usability than adding another polished micro-interaction.

Figures, pull quotes, and callouts are still present because they help pacing. They are just handled more quietly. A good post page should feel editorial, not ornamental.

Adding motion without distracting from the content

The page uses GSAP in a restrained way. The hero reveal waits until the loader finishes, the feature image picks up a slight scroll parallax, and structural elements like section headings and figures reveal as they enter the viewport. The paragraphs themselves do not animate on scroll, because that would make the act of reading feel unstable.

A small reading-progress bar helps orientation on longer posts without taking over the interface. It is one of the few motion elements that earns its place because it directly improves UX. 

const readingProgress = gsap.fromTo(
  ".reading-progress-bar",
  { scaleX: 0 },
  {
    scaleX: 1,
    ease: "none",
    scrollTrigger: {
      trigger: "#post-article",
      start: "top top",
      end: "bottom bottom",
      scrub: true
    }
  }
);

Motion is most useful on a reading page when it helps the reader stay oriented, not when it asks to be admired.

What to edit when you duplicate this file

The workflow is intentionally simple. Copy this file to blog/your-slug.html, update the title, metadata, hero image, and article content, then change the related-post links at the bottom. The CSS and JS should remain generic so every post inherits the same typography, spacing, and motion system.

  • Edit the document title and meta description in the head.
  • Replace the hero image and any inline figures with real post assets.
  • Update the tag pills, publish date, and read time in the hero.
  • Swap the sample sections out for the real article content.
  • Point the related cards at other posts once those routes exist.
A template only becomes useful once it reduces future work. The goal here is that every new post becomes an HTML editing task rather than a CSS or JavaScript task.

Keeping the finish level consistent

The final check should always be the same: does the page still feel like the rest of the site, and can you comfortably read it from top to bottom on a laptop or phone? If the answer to the second question drops, the first one no longer matters.
